# GitHub Actions Guide

This guide explains how to read and report GitHub Actions status for **BetbetMiro Extension**.

Use it when validating builds, releases, generated artifacts, `repo.json`, `plugins.json`, or workflow failures.

---

## 1. Actions Status Is Separate From Local Build

Local Gradle and GitHub Actions are different evidence points.

Use separate status lines:

```text
Gradle build lokal: SUCCESS / gagal / belum dijalankan
GitHub Actions: sukses / gagal / belum dijalankan / belum diverifikasi
```

Do not claim Actions success from a local build.

Do not claim local build success from a successful Actions run unless local Gradle was actually run.

---

## 2. What To Check In A Workflow Run

For each relevant workflow run, check:

- Workflow name.
- Branch.
- Commit SHA.
- Trigger event.
- Run status.
- Failed job, if any.
- Failed step, if any.
- Logs around the failure.
- Artifacts generated by the run.

A green workflow means the workflow completed, but it does not automatically prove runtime playback.

---

## 3. Common Actions Failures

### Gradle build failure

Usually caused by:

- Kotlin compile error.
- CloudStream API mismatch.
- Wrong import or missing helper.
- Broken provider module.
- Java or Gradle setup issue.

### Metadata generation failure

Usually caused by:

- Invalid provider `build.gradle.kts` metadata.
- Bad version value.
- Missing plugin output.
- Invalid generated JSON.

### Artifact upload failure

Usually caused by:

- Wrong output path.
- Missing artifact files.
- Workflow step order issue.
- Permission issue.

### Release or deploy failure

Usually caused by:

- Missing token permissions.
- Branch mismatch.
- Tag/release conflict.
- Invalid upload target.

---

## 4. Artifacts

When a workflow produces artifacts, check:

- Artifact name.
- Artifact size.
- Artifact upload step success.
- Whether expected plugin/build files are inside.

Do not claim an artifact exists unless it was generated or listed by the workflow.

---

## 5. repo.json and plugins.json

If Actions updates or validates repository metadata, check:

```text
repo.json: valid / invalid / belum dicek / tidak disentuh
plugins.json: valid / invalid / belum dicek / tidak disentuh
```

Actions success alone is not enough if the workflow does not validate the specific metadata you are claiming.

---

## 6. Provider Runtime Is Not Proven By Actions

GitHub Actions can prove build/workflow success.

It cannot prove CloudStream runtime behavior unless the workflow includes a real runtime validation step, which is uncommon.

Still report runtime separately:

```text
Homepage category cards: terbukti / belum terbukti / tidak relevan
Load detail/episode: terbukti / belum terbukti / tidak relevan
Playback callback link > 0: terbukti / belum terbukti / tidak relevan
```

---

## 7. Documentation-Only Workflow Notes

For documentation-only changes, a workflow may not be necessary.

Use honest wording:

```text
GitHub Actions: belum dijalankan / belum diverifikasi
Gradle build lokal: tidak dijalankan, dokumentasi saja
Provider runtime: tidak terpengaruh
```

Do not force provider validation for pure documentation commits.

---

## 8. Actions Report Template

Use this template when reporting Actions status:

```text
Workflow: ...
Branch: ...
Commit: ...
Trigger: ...
Run status: success / failed / cancelled / skipped / unknown
Failed job: ...
Failed step: ...
Artifacts: present / missing / not checked / not applicable

Gradle build lokal: SUCCESS / gagal / belum dijalankan
repo.json: valid / invalid / belum dicek / tidak disentuh
plugins.json: valid / invalid / belum dicek / tidak disentuh
GitHub Actions: sukses / gagal / belum dijalankan / belum diverifikasi
Provider runtime: terbukti / belum terbukti / tidak relevan
```

---

## 9. Maintainer Rule

A workflow result is evidence for that workflow only.

Do not extend the claim beyond what the run actually checked.
