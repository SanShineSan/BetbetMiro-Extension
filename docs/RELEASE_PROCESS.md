# Release Process

This document describes the recommended release and publishing process for **BetbetMiro Extension**.

It is focused on CloudStream extension repository maintenance, build validation, `repo.json`, `plugins.json`, GitHub Actions, and honest release status reporting.

---

## Release Priority

A release should prioritize:

1. Provider behavior remains usable.
2. Gradle build succeeds.
3. `repo.json` remains valid.
4. `plugins.json` is generated correctly.
5. GitHub Actions succeeds.
6. Documentation reflects the release accurately.

Do not prioritize cosmetic changes over provider stability.

---

## Before Releasing

Before publishing or announcing a release, check:

- Provider changes are intentional and scoped.
- Changed provider versions were bumped in their `build.gradle.kts` files.
- Documentation-only changes did not accidentally touch provider files.
- Gradle build status is known.
- Repository metadata impact is known.
- GitHub Actions status is known or clearly marked as unverified.

---

## Local Build

Use the repository Gradle wrapper.

Linux/macOS/Termux:

```bash
./gradlew make
```

Windows:

```bat
.\gradlew.bat make
```

Do not claim local build success unless the command finishes successfully.

If build fails, record:

- Failed task.
- Failed module/provider.
- Relevant error lines.
- Java version.
- OS/environment.

---

## Provider Version Bump

Every provider code change should bump that provider version in its `build.gradle.kts`.

Documentation-only changes do not need provider version bumps.

If multiple providers are changed in one batch, each changed provider should receive its own version bump.

This helps CloudStream and repository consumers detect updated plugin artifacts.

---

## repo.json

`repo.json` describes the CloudStream extension repository.

When reviewing release readiness, check:

- It exists in the expected location.
- It points to the correct plugin list URL.
- It uses valid JSON.
- It matches the branch/output strategy used by the repository.

Do not claim `repo.json` is valid unless it was checked.

---

## plugins.json

`plugins.json` is expected to represent generated plugin metadata.

When reviewing release readiness, check:

- It exists or is generated by the workflow.
- It contains expected updated providers.
- Version numbers reflect changed providers.
- Artifact URLs or download paths are correct.
- It remains valid JSON.

Do not claim `plugins.json` is valid unless it was checked.

---

## GitHub Actions

GitHub Actions should be treated as the remote build/publish authority.

Before calling a release successful, verify:

- Workflow run completed.
- Build job succeeded.
- Artifacts were created when expected.
- Plugin metadata was generated when expected.
- Release or output branch update completed when expected.

If Actions was not checked, say:

```text
GitHub Actions: belum diverifikasi
```

---

## Runtime Provider Validation

For provider releases, build success alone is not enough.

Record runtime status where possible:

```text
Homepage category cards: terbukti / belum terbukti
Load detail/episode: terbukti / belum terbukti
Playback callback link > 0: terbukti / belum terbukti
```

If runtime/device playback is not directly proven, use this wording:

```text
Playback belum terbukti di runtime/device, tetapi resolver sudah disesuaikan dengan evidence source.
```

If HAR/log evidence was used:

```text
Playback belum terbukti di runtime/device, tetapi resolver sudah disesuaikan dengan evidence source/HAR.
```

---

## Release Notes

Release notes should be honest and specific.

Good release notes include:

- Changed providers.
- Root cause summary.
- Version bumps.
- Build status.
- Runtime validation status.
- Known unverified areas.

Avoid vague claims such as:

- "All fixed" without proof.
- "Playback fixed" without callback evidence.
- "Build green" without Gradle/Actions result.
- "Repo valid" without checking metadata.

---

## Documentation-Only Release

For documentation-only changes, it is acceptable to state:

```text
Gradle build lokal: tidak dijalankan, dokumentasi saja
repo.json: tidak disentuh
plugins.json: tidak disentuh
Provider runtime: tidak terpengaruh
```

Do not run unnecessary provider edits for documentation-only releases.

---

## Manual ZIP / No Commit Mode

If a task is requested as a ZIP/manual-push artifact:

- Do not commit.
- Do not push.
- Package only the requested changed provider/module/files.
- Do not include unrelated repository files unless requested.
- State clearly that the ZIP is manual-push only.

This keeps review and deployment under maintainer control.

---

## Commit Message Format

Recommended format:

```text
fix(provider-name): short technical summary
```

For documentation:

```text
docs(scope): short summary
```

For repository maintenance:

```text
chore(scope): short summary
```

A provider fix commit body should mention:

- Root cause.
- Exact fix.
- Changed files.
- Version bump.
- Validation status.

---

## Final Release Status Template

Use this template when reporting release readiness:

```text
Changed files:
- ...

Kotlin syntax sanity: OK / gagal / belum dicek
Gradle build lokal: SUCCESS / gagal / belum dijalankan
repo.json: valid / invalid / belum dicek / tidak disentuh
plugins.json: valid / invalid / belum dicek / tidak disentuh
GitHub Actions: sukses / gagal / belum dijalankan / belum diverifikasi
Homepage category cards: terbukti / belum terbukti / tidak relevan
Load detail/episode: terbukti / belum terbukti / tidak relevan
Playback callback link > 0: terbukti / belum terbukti / tidak relevan
Commit/PR: dibuat / belum dibuat
File lain tidak disentuh: ya / tidak
```

---

## Maintainer Rule

A release is only as strong as its evidence.

If something was not checked, mark it as not checked. Honest partial status is better than unsupported claims.
